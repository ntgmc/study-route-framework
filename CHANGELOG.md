# Changelog

Release changelogs are generated by CI.

Published release notes live in GitHub Releases. The release workflow asks GitHub to generate the changelog from merged pull requests and commits, using `.github/release.yml` for category rules.

## Release Workflow

1. Update `package.json` to the release version.
2. Merge the release-ready changes into `main` or `master`.
3. Create a tag that matches the package version, such as `v2.0.0`.
4. Push the tag, or run the `Release` workflow manually with the same tag name.
5. CI runs typecheck, tests, build, E2E, and dependency audit.
6. CI generates the changelog and creates or updates the GitHub Release.

## Changelog Categories

Pull request labels control generated release-note sections:

- `breaking-change` or `breaking`
- `feature`, `enhancement`, or `feat`
- `bug` or `fix`
- `security`
- `documentation` or `docs`
- `test` or `tests`
- `chore`, `maintenance`, or `dependencies`

Use `ignore-for-release` for changes that should not appear in generated release notes.
