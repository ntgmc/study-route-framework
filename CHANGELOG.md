# Changelog

Release changelogs are generated by CI from commit history.

Published release notes live in GitHub Releases. The release workflow reads commits between the previous release tag and the new release tag, then groups them by conventional commit prefix. This works for both pull request merges and direct commits.

## Release Workflow

1. Update `package.json` to the release version.
2. Merge the release-ready changes into `main` or `master`.
3. Create a tag that matches the package version, such as `v2.0.0`.
4. Push the tag, or run the `Release` workflow manually with the same tag name.
5. CI runs typecheck, tests, build, E2E, and dependency audit.
6. CI generates the changelog from commit messages and creates or updates the GitHub Release.

## Changelog Categories

Commit message prefixes control generated release-note sections:

- `feat:` or `feature:`
- `fix:`, `bugfix:`, or `hotfix:`
- `security:`
- `docs:` or `doc:`
- `test:` or `tests:`
- `chore:`, `ci:`, `build:`, `perf:`, `refactor:`, `style:`, `deps:`, or `maintenance:`

Use `!` before the colon or include `BREAKING CHANGE` in the subject for breaking changes, for example `feat!:` or `refactor(api)!:`. Messages that do not match a known prefix are grouped under `Other Changes`.
